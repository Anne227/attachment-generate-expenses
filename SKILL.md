# 附件生成费用

将附件拆分为可追溯的独立费用，通过每刻 MCP 快速保存，并用数量与金额闭环校验结果。

## 执行原则

- 优先使用每刻 MCP；不要在常规执行中通过网页重复录入。
- 将附件中的每个收费行分别创建为费用，除非用户明确要求合并。
- 在写入前解决表单关键字段疑点；不要猜测公司抬头、往来单位、金额、收款账户或费用类型。
- 将公司抬头作为普通表单字段处理，匹配和待确认逻辑与往来单位一致；根据当前附件内容查询，不要沿用其他附件或历史任务中的字段值。
- 同时收到缴款通知书和发票时，先完成附件配对和明细核对，再决定业务场景；不要仅因存在发票就认定已经到票。
- 对非关键的数量、单价、读数等疑点，可保留附件原文到备注并在结果中标注。
- 使用来源编号和明细序号实现幂等；重试前先反查账本，避免重复创建。
- 目标执行时间为 2–4 分钟。避免重复查询、重复试错和输出完整的大型 MCP 对象。

## 工作流

### 1. 预检授权与环境

1. 检查 agent 授权状态；失效时只重新建立 MCP 会话，不要重复让用户授权，除非认证确实无效。
2. 获取当前用户、责任人、部门和企业上下文。
3. 查询表单结构，再并行查询公司抬头、往来单位、费用类型等字段的候选值。
4. 复用本次会话内已经验证的稳定技术配置，不要重复查询同一数据；业务字段值必须根据当前附件重新匹配。
5. 不要把 agent token、会话 ID 或用户私密标识写入技能文件或最终答复。

### 2. 识别并拆分附件

1. 先将附件分类为缴款通知书、发票或其他凭证，再按业务主体、往来单位、账期和金额将相关附件分组。
2. 提取缴款通知书表头：来源编号、账期、附件中的主体名称、收款单位、合同号、币种和总额。
3. 提取发票表头和明细：发票号码、开票日期、购买方、销售方、项目名称、不含税金额、税额和价税合计。
4. 按缴款通知书的收费行拆分明细，保留名称、费用所属年月、数量、单价、金额和备注。
5. 计算所有收费明细金额之和，与通知书总额核对；不一致时返回识别步骤检查，不要写入任何费用。
6. 为每条明细生成稳定来源键：`来源编号 + 明细序号 + 金额`。没有来源编号时使用文件名或附件摘要代替。

### 3. 匹配缴款通知书和发票

1. 只有缴款通知书、没有对应发票时，将费用业务场景设为“预付未到票”。
2. 同时存在缴款通知书和发票时，逐组核对：发票购买方与通知书主体相符、发票销售方与通知书收款单位相符、发票项目与收费项目业务实质相符、发票价税合计与对应缴款金额相等。多张发票对应一条或多条收费明细时，同时核对发票合计和明细分配关系。
3. 上述字段全部匹配时，将对应费用业务场景设为“到票全部支付”，并把发票附件上传并关联到费用表单；备注写入发票号码和匹配关系。
4. 金额、购销双方、费用项目或明细分配存在差异时，不要认定为到票全部支付；集中说明差异并请用户确认。
5. 写入前同时按来源键和发票号码检查重复，避免同一通知书或同一发票重复创建费用。
6. “到票全部支付”要求发票附件成功关联。附件上传或关联失败时不要把费用保存为已到票场景；先反查是否产生草稿或费用，再修复或请求浏览器降级授权。

### 4. 匹配关键字段

为每批费用确认表单中的以下字段：

- 公司抬头或法人实体
- 往来单位
- 责任人
- 业务场景，例如预付未到票、到票支付
- 费用类型
- 金额与币种
- 收款账户（任务要求付款或创建报销单时）

费用名称没有完全对应的系统类型时，按业务实质映射，并在备注中保留附件原始名称。不要为了减少费用条数而合并不同收费行。

### 5. 处理待确认项

先通过 MCP 搜索，再决定是否询问：

1. 唯一候选且名称高度一致：自动采用。
2. 多个合理候选：集中列出候选名称及可区分信息，请用户一次选择。
3. 没有候选：暂停整批写入，说明未找到的字段，请用户提供系统准确名称或决定是否先新增主数据。
4. 关键字段冲突：说明附件值、系统值和冲突点，不要自行覆盖。
5. 多个疑点同时存在：合并成一次简短确认，避免逐条打断。

关键字段未确认时，不要先创建部分费用。非关键字段不明确时，可继续创建，将原文写入备注并在结果中披露。

推荐确认话术：

> 未找到完全匹配的“附件单位名称”。系统中有以下候选：1. 候选甲；2. 候选乙。请确认使用哪个往来单位。本批费用尚未写入账本。

### 6. 构造费用对象

1. 查询当前费用类型对应的表单结构，不要凭记忆拼造必填字段。
2. 为每条明细分别构造费用对象。
3. 根据缴款通知书与发票的匹配结果写入“预付未到票”或“到票全部支付”；到票全部支付必须同时写入发票信息并关联发票附件。
4. 写入费用所属年月、数量、单价及附件备注；字段不存在时写入备注。
5. 备注至少包含来源编号、明细序号和附件原始收费名称。
6. 如果 MCP 提供附件上传能力，上传原附件并关联；若没有，明确说明费用已根据附件创建，但附件本身未上传。

### 7. 快速且安全地保存

1. 写入前按来源键查询账本，排除已经存在的明细。
2. 使用当前 MCP 中已经验证可创建费用的接口和完整标准费用结构。
3. 在每刻 MCP 的 `save_expense` 链路异常、而 `save_expense_detail` 已被验证可用时，使用 `save_expense_detail` 创建新费用；不要对同一故障接口反复试错。
4. 第一条成功后确认返回的费用编号、金额、状态和关键自定义字段。
5. 其余明细按 3–4 条有限并发保存；每条使用不同来源键。
6. 只提取保存响应中的 `success`、费用编号、金额、状态和错误，不要把完整费用对象送回上下文。
7. 出现系统性错误时停止发起新写入；出现单条数据错误时记录该条并继续处理明确安全的条目。
8. 任何重试都先反查来源键，确认前一次请求没有实际落账。

### 8. 闭环验证

保存完成后使用 MCP 按来源编号反查账本，并同时验证：

- 费用数量等于附件明细数量
- 金额合计等于附件总额
- 每条状态符合预期，通常为“未报销”
- 没有重复来源键
- 公司抬头、费用类型和往来单位等表单字段正确
- 业务场景符合附件组合；到票全部支付的费用已关联正确发票号码和发票附件

验证失败时，只修正缺失或错误的条目；不要整批重新创建。

## 结果反馈

最终简洁报告：

- 创建条数和金额合计
- 账本状态
- 按费用类型汇总的金额
- 公司抬头、往来单位和业务场景
- 未确认、未上传或降级记录的字段
- 是否与已有报销单关联

不要宣称成功，除非已经从账本反查到对应费用并完成数量和金额校验。

## 浏览器降级规则

MCP 能完成任务时不要使用浏览器。只有 MCP 缺少必要能力或持续返回系统错误，并且用户明确同意浏览器降级后，才通过已登录页面继续；浏览器操作仍需遵守写入确认与幂等检查。
